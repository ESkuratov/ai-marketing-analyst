# ADR-001: Выбор архитектуры marketing-analyst агента

**Дата:** 2026-05-03
**Статус:** Accepted
**Контекст:** ADR для AI-агента `marketing-analyst` сети массажных студий

---

## Характер работы агента (ключ к выбору)

- **Batch-обработка по расписанию** — daily/weekly/monthly, не real-time
- **Pipeline линейный** — S1→S2→S3→S4→S5, без ветвлений и сложной координации
- **Данные уже персистентны** — лиды в AMO, записи в YClients, расходы в Google Sheets
- **Метрики оконные** — конверсия за день/неделю/месяц, CAC скользящий, WoW/MoM тренды
- **Операции idempotent** — повторный запуск за тот же период даёт тот же результат

---

## Рассмотренные альтернативы

### 1. Stateless + Shared Memory (текущее решение) ✅

**Почему оптимально:**

| Характеристика | Как работает | Почему подходит |
|---|---|---|
| Запуск по cron | Каждый запуск — чистый процесс | Работа и так пакетная, real-time не нужен |
| Idempotent pipeline | UPSERT по `studio_id + id` | Можно перезапускать без последствий |
| Shared memory (PG) | Все данные в одной БД | Агент и так всё пишет в PG, дублирования нет |
| Multi-studio | `studio_id` параметризует pipeline | Один код на все студии |
| Отказоустойчивость | Упал — перезапустился с того же шага | Не нужно управлять состоянием сессии |
| Debug | Каждый запуск изолирован | Логи + данные в PG = полная воспроизводимость |

**Главное:** архитектура совпадает с характером работы. Агент — это pure function: `(cron_trigger, api_data) → (metrics_in_db, report_in_telegram)`. Stateless не добавляет ограничений, а убирает необходимость управлять состоянием.

---

### 2. Stateful (in-memory state)

**Что предлагает:** агент держит кэш, пулы соединений, текущие метрики в памяти между запусками.

**Проблемы:**
- **Дублирование состояния** — все данные уже в PostgreSQL. Кэш в памяти = второй источник истины с риском рассинхрона
- **Crash = потеря** — агент упал → потерян кэш, соединения, промежуточные результаты. Для batch-обработки это чистая потеря
- **Масштабирование** — второй инстанс агента не имеет доступа к памяти первого. Придётся externalize state → возвращаемся к shared memory
- **Нет выигрыша** — batch-обработка не выигрывает от in-memory кэша: каждый раз читаются данные за новый период, которые всё равно нужно выбирать из API или PG

**Когда stateful оправдан:** когда агент держит live-соединение (WebSocket), обрабатывает потоковые данные или ведёт диалог. У marketing-analyst ничего этого нет.

---

### 3. Event-Driven (реакция на события вместо cron)

**Что предлагает:** новый лид в AMO → событие → запуск S1 → и т.д.

**Проблемы:**
- **Метрики оконные** — конверсия "лид→запись" за день не считается по одному лиду. Нужен batch по периоду. События не отменяют batch, они его усложняют
- **Сверка (Reconcile)** — сравнение AMO↔YC по одному лиду бессмысленно, нужна полная выборка за период
- **Event ordering** — лид может прийти раньше, чем связанная запись в YClients. Out-of-order events = вечная головная боль
- **Exactly-once** — потеря события = потеря лида. Доставка без гарантий невозможна для финансовых данных
- **Дополнительная инфраструктура** — нужен event bus (Kafka/RabbitMQ), DLQ, retry механизмы

**Итог:** Event-driven добавляет complexity без выгоды, потому что:
- Источник правды — API (AMO, YClients), где события уже есть в виде записей с датами
- Все метрики считаются за окно (день/неделя/месяц) — batch неизбежен
- Reconciliation требует полного сравнения, а не событий

---

### 4. Dispatcher (отдельный оркестратор pipeline)

**Что предлагает:** отдельный сервис, который управляет запуском шагов, retry, расписанием, блокировками.

**Проблемы:**
- **Over-engineering** — pipeline из 5 линейных шагов не требует оркестратора. Нет ветвлений, параллельных шагов, условной логики
- **Ещё одна сущность** — нужно деплоить, мониторить, обеспечивать HA
- **Cron уже есть** — OpenCLAW Cron Scheduler + idempotent pipeline = достаточное управление
- **HITL gates** — единственная "условная логика" (пропуск студии если gate открыт) решается одним SELECT в pipeline

**Когда dispatcher оправдан:** когда pipeline содержит сложную логику (parallel steps, conditional branching, human approval workflows с разветвлениями). Здесь этого нет.

## Сводная таблица

| Критерий | Stateless + Shared Memory | Stateful | Event-Driven | Dispatcher |
|---|---|---|---|---|
| Сложность реализации | ★☆☆ (низкая) | ★★☆ | ★★★ | ★★★ |
| Отказоустойчивость | ★★★ (перезапуск) | ★☆☆ (crash = loss) | ★★☆ (нужен replay) | ★★☆ (SPOF) |
| Масштабирование | ★★★ (studio_id) | ★☆☆ (shared state) | ★★☆ (партиции) | ★★☆ |
| Соответствие workload | ★★★ (batch) | ★★☆ | ★☆☆ (оконные метрики) | ★☆☆ (линейный pipeline) |
| Debug | ★★★ (изолированные запуски) | ★☆☆ (зависит от истории) | ★★☆ (trace across events) | ★★☆ |
| Операционные расходы | ★☆☆ (минимальные) | ★★☆ | ★★★ (event bus) | ★★★ |

## Решение

**Stateless + Shared Memory** — не просто "хорошее решение", а единственное, которое совпадает с характером работы агента. Остальные подходы либо добавляют complexity без выгоды (event-driven, dispatcher), либо конфликтуют с batch-природой задач (stateful).
